# Project Brief

Last updated: 2026-06-29

This is the single project truth document for the NYC TLC Green Taxi MLOps project. It replaces the older split state files `docs/project_state.md`, `docs/project_plan.md`, and `docs/notebook_structure.md`.

## Objective

Build a reproducible MLOps proof of concept using NYC TLC Green Taxi trip records. The course deliverable must show a deployment-oriented workflow, not only a modeling notebook:

- Data ingestion, profiling, validation, and documented quality findings.
- Feature engineering with a clear training-serving boundary.
- MLflow experiment tracking, model comparison, and model artifact logging.
- Kedro pipelines for stable workflows.
- Explainability artifacts.
- Serving API and Docker container.
- Drift or monitoring evaluation using later Green Taxi batches.
- Tests for implemented behavior.
- A final report of at most 6 pages with evidence generated by the project.

The project should stay understandable for a first-time MLOps student. Notebooks should teach the workflow visibly before code is extracted into reusable modules or Kedro pipelines.

## Decisions

Locked decisions:

- Dataset family: NYC TLC Green Taxi trip records.
- Package name: `mlops_project`.
- Dependency manager: `uv`.
- Core stack: Kedro, Great Expectations or simple documented checks, MLflow, Optuna after baselines, SHAP or permutation importance, FastAPI, Docker, and a justified drift method such as Evidently, PSI, JS divergence, or NannyML.
- Raw data layout: one Green Taxi partitioned source under `data/01_raw/green_taxi/`; do not duplicate raw files into separate train/drift folders.
- Current working target: `tip_amount` regression.
- Current modeling period: 2024-2025 Green Taxi data.
- Current drift holdout: available 2026 Green Taxi months.
- Current train/test split parameter: `train_test_split_date: "2025-07-01"`.
- Current train/validation split parameter: `train_val_split_date: "2025-01-01"`.
- Current catalog convention: `03_primary` stores split raw analytical tables, `04_feature` stores reusable preprocessing and selected-column artifacts, `05_model_input` stores preprocessed feature matrices consumed by model nodes, `07_model_output` stores predictions, and `08_reporting` stores metrics, reports, plots, and validation summaries.
- Hopsworks is optional and controlled by configuration. Keep credentials out of version control.
- Final notebook outputs should remain visible after reviewed reruns.

Evidence-dependent decisions that must stay explicit:

- Final production model and champion/challenger promotion decision.
- Final serving-time feature schema.
- Final cleaning and outlier thresholds.
- Final success metric thresholds.
- Drift method, alert thresholds, and whether any checks become blocking.
- Exact figures, tables, and claims included in the final report.
